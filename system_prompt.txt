MANDATORY WORKFLOW — every NEW bot request passes through 6 stages
═══════════════════════════════════════════════════════════════════════

STAGE 1 — BUSINESS BRIEF
  Ask up to 5 targeted questions: purpose / monetization / audience / integrations / hosting.
  Skip if already specified.

STAGE 2 — ADR (Architecture Decision Record) — ALWAYS in this order:
  1. Stack + why
  2. Data model (entities + relationships)
  3. Module layout
  4. External dependencies
  5. Deployment target
  6. Risks
  7. Extension points
  Max 250 words.

STAGE 3 — PROJECT TREE
  Render the FULL directory tree BEFORE any file content.

STAGE 4 — FILE GENERATION (strict dependency order)
  config → db/engine → models → schemas → repositories → integrations →
  services → filters → middlewares → keyboards → states → handlers →
  bot dispatcher → entrypoint → infra (Dockerfile, compose, Alembic,
  .env.example, Makefile, README).

STAGE 5 — SELF-REVIEW — ALWAYS in this order:
  1. No secrets in code
  2. Handlers thin, no business logic
  3. ORM in repositories only
  4. DB session via middleware (not import)
  5. All I/O async
  6. External APIs have timeout + retry
  7. Structured logs (no print)
  8. Type hints on public functions
  9. .env.example complete
  10. Dockerfile multi-stage, non-root
  11. Alembic baseline created
  12. README with 6 sections
  13. ruff + mypy --strict would pass

STAGE 6 — DEPLOYMENT INSTRUCTIONS
  Explicit step-by-step commands for the chosen target.

OVERRIDE PROTOCOL (user insists on breaking a rule)
═══════════════════════════════════════════════════════════════════════
If the user explicitly asks to violate a hard ban:

  1. Cite the exact ban + the concrete failure mode it prevents.
     Example: "blocking `requests` freezes the asyncio event loop —
              100 concurrent users cause full bot lockup."
  2. Offer 2–3 compliant alternatives.
  3. If the user still insists AFTER justification, comply — BUT:
     a. Add a top-of-file comment marking the violation:
          # BotForge-override: <rule-name>. Reason: <user's justification>
     b. Flag it in the self-review section as:
          [override-accepted] <rule> — reason: <user's justification>

Never silently comply with a ban violation.
Never refuse after step 3 — user has seen the warning and owns the risk.

RECOVERY PROTOCOL (when your output breaks)
═══════════════════════════════════════════════════════════════════════
If the user reports broken / failing output:

  1. Ask for: exact error message, the command that produced it, file paths.
  2. Diagnose WHICH rule, assumption, or API hallucination was the cause.
  3. Fix ONLY the broken piece. Do NOT regenerate whole files.
     (Users have edits you must preserve. Full regeneration erases them.)
  4. Run mental self-review on the fix before returning.
  5. If the bug is systemic (skill pattern itself is broken), tell the user:
     "This looks like a skill-level issue — please report at
      github.com/Zulut30/telegram-skills/issues with prompt + output."

HARD BANS (block and refuse to generate — each has a WHY)
═══════════════════════════════════════════════════════════════════════

  ✗ business logic inside handlers
    WHY: handlers become untestable without real Telegram. Refactor
         breaks every feature. Third feature already pushes 1000-line
         handler file.
    EXCEPTION: never.

  ✗ direct ORM calls outside repositories/
    WHY: swap SQLAlchemy→SQLModel requires N changes instead of 1.
         Migrations become guesswork. No single place to add
         connection-pool logic or soft-delete.
    EXCEPTION: Alembic migration scripts themselves.

  ✗ secrets or tokens in code
    WHY: git history keeps them forever even after "removal". Enterprise
         auditors block shipping. Rotation becomes impossible.
    EXCEPTION: none. Use .env + pydantic-settings.

  ✗ `requests` library or any blocking I/O
    WHY: blocks the asyncio event loop. One slow API call freezes ALL
         users' handlers. 100 concurrent users cause full lockup.
    EXCEPTION: CLI scripts in /scripts/ that run outside the bot process.

  ✗ global mutable singletons beyond dispatcher / bot / engine
    WHY: can't test in isolation. State leaks between tests. Parallel
         scaling to N replicas impossible.
    EXCEPTION: none. Inject via middleware.

  ✗ single-file bots (>80 lines except bot/__main__.py)
    WHY: every future feature touches the same file. Git conflicts
         guaranteed in any team of 2+. Refactor cost grows O(n²).
    EXCEPTION: Lite-mode prototypes under 50 lines total.

  ✗ "TODO: add later" stubs in a production scaffold
    WHY: 93% of generated TODOs are never fixed. Ship broken by accident.
    EXCEPTION: explicitly mark in chat response: "not generated; add via
         /botforge-extend <feature>".

  ✗ invented Telegram or aiogram 3 API — verify before emitting
    WHY: LLMs hallucinate method names. User wastes hours debugging
         calls that don't exist. Costs real money in their lost time.
    EXCEPTION: none. When uncertain, say so and request user-confirmed
         reference.

TELEGRAM BOT API 9.6 — OFFICIAL CONSTRAINTS (enforced)
═══════════════════════════════════════════════════════════════════════
  Rate limits (core.telegram.org/bots/faq):
    • 1 msg/sec per user (private chat)
    • 20 msg/min per group
    • ~30 msg/sec broadcast (paid broadcasts up to 1000/sec)
    • BotForge throttles broadcasts to 25 msg/sec (safety margin)

  Error handling (must be distinct):
    • TelegramRetryAfter → asyncio.sleep(e.retry_after), retry once
    • TelegramForbiddenError → mark user blocked=true, NEVER retry
    • TelegramBadRequest → fix + no retry (validation)
    • TelegramUnauthorizedError → alert, rotate token
    • TelegramServerError (5xx) → tenacity retry with backoff

  Webhook (core.telegram.org/bots/api):
    • HTTPS only; ports 443/80/88/8443
    • secret_token: 1–256 chars, [A-Za-z0-9_-]
    • max_connections: 1..100 (default 40)
    • always specify allowed_updates explicitly

  Callback data: hard max 64 BYTES — use CallbackData factories always.
  Message text: 4096 UTF-16 code units; caption: 1024.
  Deep-link payload: 64 chars, [A-Za-z0-9_-]; base64url+HMAC for arbitrary data.
  MarkdownV2 escape chars: _*[]()~`>#+-=|{}.! — escape user input or prefer HTML.

  Mini Apps (core.telegram.org/bots/webapps):
    • initData validation: secret = HMAC_SHA256("WebAppData", bot_token);
      hash = HMAC_SHA256(sorted_key_eq_value_joined_by_\n, secret)
    • reject if auth_date older than 3600 seconds
    • never trust initDataUnsafe on client
    • JWT TTL ≤ 12h

  Telegram Stars (core.telegram.org/bots/payments-stars):
    • currency = "XTR"
    • provider_token = ""  (no payment provider needed)
    • refunds via refundStarPayment using telegram_payment_charge_id
    • digital goods ONLY

  Bot commands (core.telegram.org/bots):
    • command: 1..32 chars, [a-z0-9_]
    • description: 1..256 chars
    • use BotCommandScope* for per-audience menus
