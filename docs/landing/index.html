Real bot. Real flow. Real code.

  • User starts the bot

    handlers/common.py receives CommandStart, calls UserService.ensure_user, returns the main menu inline keyboard.

  • Inline menu navigation

    callback_query:"menu:vip" routes to subscription.show_vip. Handler stays thin — all logic in SubscriptionService.

  • Payment invoice

    PaymentService.start_vip_purchase creates an order in Postgres, calls StarsProvider.create_invoice, returns pay URL.

  • Idempotent webhook

    successful_payment handler checks PaymentRepo.is_processed — duplicate webhooks return without double-activation.

  • Access granted

    SubscriptionService.activate extends or creates subscription, channel invite link issued, audit logged.

How it works

Three rules your AI cannot break.

BotForge enforces these through the skill prompt itself. The AI literally refuses to generate code that violates them.

01 — ARCHITECTURE

No monoliths.

Handlers, services, repositories, integrations — layered from day one. Business logic in a handler is a [blocker].

02 — PRODUCTION

No drafts.

Docker, Postgres, Alembic, .env, webhook, deploy commands — in the very first generation. Zero "TODO: add later".

03 — EXTENSIBILITY

No breakage.

Diff-aware extension protocol. Adding the fifth feature is as easy as the first — and doesn't invalidate existing tests.
